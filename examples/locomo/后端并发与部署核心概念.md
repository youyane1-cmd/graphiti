## 1. 你的服务器现在长什么样

你启动服务的方式是：

```python
if __name__ == '__main__':
    uvicorn.run(app, host=host, port=port)
```

这一行启动出来的东西，画出来是：

```text
┌─────────────────── 一个 Python 进程 ───────────────────┐
│                                                        │
│   ┌────────────── 一个事件循环 (event loop) ─────────┐  │
│   │                                                 │  │
│   │   请求A ──┐                                      │  │
│   │   请求B ──┼── 排队交替执行（不是同时执行）          │  │
│   │   请求C ──┘                                      │  │
│   │                                                 │  │
│   └─────────────────────────────────────────────────┘  │
│                                                        │
│   app.state.graphiti      ← 全局共享，所有请求都用它     │
│   app.state.user_graphiti ← 进程内的 dict 缓存          │
│   app.state.user_locks    ← 进程内的 asyncio.Lock       │
│                                                        │
└────────────────────────────────────────────────────────┘
```

三个关键事实：

1. **默认只有一个 Python 进程**。同一时刻通常只有一个线程执行 Python 字节码；
   网络 I/O、某些释放 GIL 的 C 扩展仍可能由操作系统或底层库并行处理。
2. **只有一个事件循环**。所有请求在同一个循环里"交替"跑，靠 `await` 让出执行权。
3. **状态全在进程内存里**。`app.state` 上的 dict、锁，都只存在于这一个进程的内存中。

这套架构**没有错**，它能正确服务的场景是：请求量不大、单机、允许偶尔慢。你的评测场景完全够用。
但它有明确的天花板，后面每一节讲一块天花板怎么捅破。

### 1.1 从启动到关闭：`lifespan` 和 `app.state`

`uvicorn.run(app, ...)` 不只是“调用一个函数”。它大致做了这些事：

```text
操作系统启动 Python 进程
  → uvicorn 创建事件循环
  → FastAPI 执行 lifespan 启动部分
  → 创建 Graphiti、user_graphiti、user_locks
  → 开始监听端口并接收请求
  → 每个请求被包装成 asyncio Task
  → 服务退出时执行 lifespan 的 finally，调用基础 Graphiti 客户端的 close()
```

`main_server.py` 中这三个对象的生命周期都是“一个 worker 从启动到退出”：

| 对象 | 含义 | 重启/增加 worker 后怎样 |
|------|------|-------------------------|
| `app.state.graphiti` | 基础 Graphiti 客户端 | 每个 worker 各创建一份 |
| `app.state.user_graphiti` | `user_id → Graphiti` 的进程内缓存 | 重启清空；每个 worker 各一份 |
| `app.state.user_locks` | `user_id → asyncio.Lock` | 重启清空；不同 worker 的锁互不认识 |

`app.state` 适合存“本进程共享、丢失后可重建”的对象，例如客户端和锁；不适合存余额、
任务最终状态等必须持久化的数据。多 worker 都必须看到的状态，应放 Redis 或数据库。
当前 lifespan 明确关闭了基础 `graphiti`；用户 clone 是否共享底层连接、是否还需逐个关闭，
要以驱动实现为准，扩展连接缓存时必须把这项纳入生命周期测试。

### 1.2 请求、协程和 Task 是什么关系

FastAPI 调用一个 `async def` 路由时，会得到一个**协程对象**；事件循环把它作为
`asyncio.Task` 调度。Task 记录“执行到哪里、正在等什么、结果或异常是什么”。

```text
HTTP 请求 A → Task A → 执行 register() → await LLM → 暂停
HTTP 请求 B → Task B → 趁 A 暂停时执行 search() → await DB → 暂停
LLM 返回    → Task A 被唤醒，从原来的 await 后继续
```

因此“停掉某个用户”通常不是杀进程，而是找到该用户对应的 Task 并取消，同时拒绝他的
后续请求。一个 worker 进程同时承载很多用户，杀 worker 会误伤该进程里的所有请求。

## 2. 并发的地基

### 2.1 并发（Concurrency）vs 并行（Parallelism）

| 概念 | 一句话 | 例子 |
|------|--------|------|
| 并发 | **交替**处理多件事，看起来同时 | 一个厨师轮流看 3 口锅 |
| 并行 | **真正同时**处理多件事 | 3 个厨师各看 1 口锅 |

你现在服务器的 **Python 路由主逻辑**主要是单线程协作式并发：一个事件循环轮流照看请求。
这不等于整个进程绝对没有并行；网络栈、数据库服务和某些 C 扩展可能在别处并行工作。

### 2.2 I/O 密集 vs CPU 密集

这是决定“主要采用哪种并发方式”的首要分类标准：

| 类型 | 特征 | 你的例子 |
|------|------|----------|
| I/O 密集 | 大部分时间在**等**：等网络、等数据库、等磁盘 | 等 OpenAI 返回、等 FalkorDB 查询结果——你的服务 99% 是这个 |
| CPU 密集 | 大部分时间在**算**：矩阵运算、压缩、解析 | 本地跑 embedding 模型、rerank 模型推理 |

判断口诀：

> 等的时候 CPU 是闲的 → I/O 密集 → 用协程（asyncio）最划算。
> 算的时候 CPU 是满的 → CPU 密集 → Python 工程中通常用多进程、原生扩展或独立计算服务。

你的 `/memory/add` 一次要调多次 LLM（实体抽取、去重、时序判断），每次调用要等几百 ms 到几秒——**等待期间 CPU 完全空闲**。这就是为什么 Graphiti 全库都是 `async def`：等 LLM 的时候，事件循环可以去服务别的请求。

## 3. Python 的三种并发方式

### 3.1 多线程（threading）—— 受 GIL 限制

**GIL（全局解释器锁）**：CPython 规定同一时刻只有一个线程能执行 Python 字节码。

后果：

- 在常见 CPython 版本里，多线程通常**不能**加速纯 Python CPU 密集任务
  （10 个线程算 Python 循环仍会争抢同一把 GIL）。
- 多线程**可以**加速 I/O 密集任务（线程 A 等网络时，GIL 释放，线程 B 能跑）。

最简单的验证实验（可直接运行，亲眼看 GIL）：

```python
import threading
import time

def count():
    n = 0
    for _ in range(50_000_000):   # 纯计算，不等任何东西（CPU 密集）
        n += 1

# 单线程干两遍
start = time.time()
count(); count()
print(f'单线程: {time.time() - start:.1f} 秒')     # 例如 4.0 秒

# 两个线程"同时"干
start = time.time()
t1 = threading.Thread(target=count)
t2 = threading.Thread(target=count)
t1.start(); t2.start()
t1.join(); t2.join()
print(f'双线程: {time.time() - start:.1f} 秒')     # 还是 ~4.0 秒，甚至更慢！
```

双线程没有变快，因为 GIL 让两个线程轮流拿"执行权"，本质还是一个核在算——这就是"多线程不能加速 CPU 密集"的实证。但如果把 `count` 换成"等网络 2 秒"，双线程就只要 2 秒（等待时 GIL 会释放）：

```python
# 多线程处理 I/O 是可行的，但线程有成本：
# 每个线程都有平台相关的栈和调度成本，线程切换靠操作系统，不能无限创建
import threading

def call_llm(prompt):          # 同步阻塞函数
    response = requests.post(...)   # 等待时该线程挂起，GIL 让给别人

threads = [threading.Thread(target=call_llm, args=(p,)) for p in prompts]
```

### 3.2 多进程（multiprocessing）—— 绕过 GIL

每个进程是**独立的 Python 解释器**，各有各的 GIL，各有各的内存。

- 优点：多份解释器可以真并行，能吃满多核，是纯 Python CPU 密集任务最常见的方案。
- 缺点：**内存不共享**（这是第 6 节所有问题的根源）、启动慢、进程间通信要序列化。

### 3.3 协程（asyncio）—— 你正在用的

协程是**用户态的轻量任务**：不靠操作系统切换，靠代码里的 `await` 主动让出。

```python
async def register(request, http_request):
    ...
    await graphiti.add_episode(...)   # ← 走到这里，"我要等 LLM 了"
                                       #    事件循环立刻去跑别的请求
                                       #    LLM 返回后，从这里继续
```

三种方式对比：

| | 线程 | 进程 | 协程 |
|---|------|------|------|
| 切换者 | 操作系统 | 操作系统 | 事件循环（代码里的 await） |
| 单个成本 | ~8MB | ~几十MB | ~几KB |
| 能开多少 | 几百个 | 核数级别 | 几万个 |
| 加速 CPU 密集 | ✗（GIL） | ✓ | ✗ |
| 加速 I/O 密集 | ✓ | ✓（浪费） | ✓（最优） |
| 共享内存 | 共享（要加锁） | 不共享 | 共享（仍可能在 `await` 处发生竞态） |

**结论**：LLM 应用后端 = I/O 密集为主 → asyncio 是主力，多进程（worker）用来利用多核，线程只在"被迫调用同步库"时垫底用。你的技术选型（FastAPI + async）从一开始就是对的。

### 3.4 asyncio 零基础十分钟（最简单的可运行例子）

上面讲的都是"是什么"，这一小节用一个能直接复制运行的例子讲"怎么回事"。

**第一步：先看普通同步代码的问题。**

场景：泡茶。烧水要 3 秒，洗杯子要 2 秒。

```python
import time

def boil_water():
    print('开始烧水...')
    time.sleep(3)          # 假装在等水开（等待期间人干站着）
    print('水开了')

def wash_cups():
    print('开始洗杯子...')
    time.sleep(2)
    print('杯子洗好了')

start = time.time()
boil_water()
wash_cups()
print(f'总耗时 {time.time() - start:.1f} 秒')
```

运行结果：

```text
开始烧水...
水开了
开始洗杯子...
杯子洗好了
总耗时 5.0 秒
```

问题显而易见：**烧水的 3 秒里你在干站着等**，明明可以趁这时间去洗杯子。5 秒 = 3 + 2，纯浪费。

**第二步：asyncio 版本——等水开的时候去洗杯子。**

```python
import asyncio
import time

async def boil_water():
    print('开始烧水...')
    await asyncio.sleep(3)     # "我要等 3 秒，这期间你们谁想干活谁干"
    print('水开了')

async def wash_cups():
    print('开始洗杯子...')
    await asyncio.sleep(2)
    print('杯子洗好了')

async def main():
    start = time.time()
    await asyncio.gather(boil_water(), wash_cups())   # 两件事一起跑
    print(f'总耗时 {time.time() - start:.1f} 秒')

asyncio.run(main())
```

运行结果：

```text
开始烧水...
开始洗杯子...          ← 注意！没等水开就开始洗杯子了
杯子洗好了
水开了
总耗时 3.0 秒          ← 不是 5 秒
```

**第三步：逐个关键词解释（就 4 个）。**

| 关键词 | 意思 | 类比 |
|--------|------|------|
| `async def` | 声明"这个函数中途可以被暂停" | 这道菜做到一半可以先放着 |
| `await` | "我要开始等了，把执行权让出去；等的事完成后再从这里继续" | 把水壶插上电，人走开去干别的 |
| `asyncio.gather(a(), b())` | 把多个任务同时挂到事件循环上一起跑 | 同时开工几件事 |
| `asyncio.run(main())` | 启动事件循环（那个"总调度员"），跑完 main 为止 | 厨房开工 |

**最容易误解的一点**：`await` 不是"卡在这里等"，而是"**让**"。执行到 `await asyncio.sleep(3)` 时，这个函数暂停、控制权交回事件循环，事件循环马上去跑别的任务（洗杯子）；3 秒后水开了，事件循环再回来从暂停的地方继续。**从头到尾只有一个线程**，只是它从不干等。

**第四步：对应到你的服务器。**

把例子里的词换掉，就是你的 main_server.py：

```text
烧水 3 秒          →  alice 的 add_episode 在等 OpenAI 返回（1-2 秒）
洗杯子 2 秒        →  bob 的 /memory/search 请求
事件循环           →  uvicorn 帮你启动的那个循环
```

alice 的请求在 `await`（等 LLM）时，事件循环转头就去处理 bob 的请求——所以你的单进程服务器能同时服务多个用户，这就是 asyncio 送你的并发。

**第五步：两个新手必踩的坑。**

```python
# 坑 1：async 不会让单个任务变快
await asyncio.sleep(3)    # 该等 3 秒还是 3 秒，快的是"等的时候能干别的"

# 坑 2：在 async def 里用了同步的 time.sleep
async def bad():
    time.sleep(3)     # 灾难！这不会"让"，整个事件循环干等 3 秒
                      # 期间所有其他请求全部卡住（第 4.2 节细讲）
    # 正确写法： await asyncio.sleep(3)
```

判断口诀：**async 函数里凡是"要等"的操作（网络、数据库、sleep），必须用带 await 的异步版本**；用了同步版本，等价于厨师守着水壶不许任何人做任何事。

### 3.5 到底哪些函数写 `async def`，哪些地方写 `await`

先记住：`async` 不是“让函数自动变快”的装饰器，`await` 也不能随便加。判断从你调用的
**最底层库**开始，沿调用链向上传播。

#### 规则 1：调用异步函数，当前函数也必须是异步函数

```python
async def save_episode(graphiti, message):
    # add_episode 是 async 函数，调用它必须 await
    await graphiti.add_episode(...)

async def register(...):
    # save_episode 也是 async，上一层继续 await
    await save_episode(graphiti, message)
```

如果忘记 `await`：

```python
result = graphiti.add_episode(...)  # 没有真正执行，只得到 coroutine 对象
```

常见结果是 `RuntimeWarning: coroutine was never awaited`，数据根本没有写进去。

#### 规则 2：等待 I/O 才是 async 的主战场

| 操作 | 推荐方式 | 原因 |
|------|----------|------|
| 调 OpenAI/httpx 异步客户端 | `await client.post(...)` | 等网络时让事件循环服务别人 |
| 调异步数据库/Redis | `await driver.query(...)` | 等远端响应 |
| 异步休眠 | `await asyncio.sleep(...)` | 不阻塞循环 |
| 几行字符串拼接、字典查询 | 普通同步代码 | 太快，不值得切换 |
| 很小的配置/进度文件 | 可暂时同步，但要监控耗时 | 简单；文件变大后会阻塞 |
| 同步慢库或慢磁盘 | `await asyncio.to_thread(fn, ...)` | 把阻塞移到线程池 |
| 大量纯 Python 计算 | 进程池/独立 worker | async 不会让计算并行 |

不是所有辅助函数都要加 `async`。例如只做正则替换的 `register_progress_path()` 没有任何
等待点，保持普通 `def` 更清楚。只有当函数内部需要 `await`，它才必须写成 `async def`。

#### 规则 3：顺序有依赖就逐个 `await`，互不依赖才 `gather`

```python
# B 依赖 A 的结果：必须顺序
entities = await extract_entities(text)
edges = await build_edges(entities)

# 两个只读查询互不依赖：可以并发等待
nodes, edges = await asyncio.gather(
    search_nodes(query),
    search_edges(query),
)
```

`main_server.py` 的 `search_context()` 同时查询节点和边，就是 `gather` 的正确例子。
而 `/memory/add` 中多条 episode 有时序和去重依赖，所以当前选择逐条 `await add_episode()`，
不能为了“快”直接把整批消息全部 `gather`。

#### 规则 4：`await` 是可能发生任务切换的位置

从一个 `await` 到下一个 `await` 之间，事件循环不会主动切到另一个普通协程。但不要把它
误解成“绝对原子”：你调用的同步代码可能涉及线程、进程或数据库并发。共享状态跨越
`await` 的“读 → 等 → 写”过程时，应重新检查是否需要锁。

快速判断流程：

```text
这个函数内部需要调用 async API 吗？
├─ 是 → 写 async def，并在调用点 await
└─ 否
   ├─ 只做很快的内存计算 → 普通 def
   ├─ 调同步 I/O/阻塞库 → 普通 def 交给 FastAPI 线程池，
   │                       或在 async 函数中 asyncio.to_thread
   └─ 做重 CPU 计算 → 进程池或独立计算 worker
```

### 3.6 GIL、事件循环和 worker 不要混在一起

- **GIL** 管“一个 CPython 解释器里，线程何时能执行 Python 字节码”。
- **事件循环**管“很多协程在什么时机暂停和恢复”。
- **worker** 是独立进程，每个进程有自己的解释器、GIL、事件循环和内存。
- `asyncio` 通过等待 I/O 时切换任务提升吞吐，不会绕过 GIL 加速纯 Python 计算。
- 多 worker 能利用多核，但会带来状态分裂、连接数增加和跨进程协调问题。

## 4. FastAPI 怎么处理请求

### 4.1 async def 和 def 的天壤之别

FastAPI 对两种写法的处理方式完全不同：

```python
@app.post('/a')
async def handler_a():        # 在事件循环里跑
    await something()          # 必须用 await 让出，否则会卡住整个循环

@app.post('/b')
def handler_b():               # FastAPI 自动扔进线程池跑（容量由运行时配置，通常几十量级）
    time.sleep(10)             # 阻塞的是线程池里的一个线程，不影响事件循环
```

### 4.2 最经典的事故：在 async def 里写同步阻塞代码

```python
@app.post('/bad')
async def bad_handler():
    time.sleep(10)             # 灾难：事件循环被卡死 10 秒
                               # 期间【所有】请求都无响应，包括 /health
```

为什么灾难：事件循环是单线程的，`time.sleep` 不会让出执行权，整个循环停摆。

你的代码里有一个轻微版本的这个问题：

```python
# main_server.py 里的 register 接口（async def 内部）：
group_progress = load_register_progress(group_id)   # 同步读文件
append_register_progress(group_id, progress_item)    # 同步追加一行 JSONL
```

`json.load`/`json.dump` 读写小文件是毫秒级，所以现在没感觉。但要建立这个意识：**async def 里的每一行同步代码，执行期间全服务器都在陪它等**。文件一大或磁盘一慢，所有用户一起卡。

补救手段（按优先级）：

```python
# 1. 换异步库（最正规）
import aiofiles

# 2. 扔线程池（最省事，适合改不动的同步库）
import asyncio
progress = await asyncio.to_thread(load_register_progress, group_id)
```

### 4.3 你的接口现在的实际并发行为

以 `/memory/add` 为例，梳理一次完整请求：

```text
请求进来
  → async with get_user_lock(...)      # 拿这个 user 的锁
  → 逐条消息 await add_episode(...)     # 每条内部有多次 LLM 调用，秒级
  → 全部完成，返回

关键点 1：同一个 user 的两个请求会排队（锁保证），这是对的——
         Graphiti 写图有先后依赖，并发写同一个图会乱。
关键点 2：不同 user 的请求【可以】并发——A 用户在等 LLM 时，
         事件循环会去处理 B 用户的请求。这是 asyncio 白送你的。
关键点 3：但一次 add 要处理完【所有】消息才返回，客户端要干等几十秒
         甚至几分钟——这就是第 8 节消息队列要解决的问题。
```

### 4.4 你代码里的 per-user 锁：加和不加到底差在哪

你的代码里有这两段（AI 帮你写的，现在把它讲透）：

```python
def get_user_lock(request: Request, user_id: str) -> asyncio.Lock:
    locks: dict[str, asyncio.Lock] = request.app.state.user_locks
    return locks.setdefault(user_id, asyncio.Lock())

# 接口里这样用：
async with get_user_lock(http_request, request.user_id):
    ...  # 处理 alice 的数据
```

**先用一个最简单的例子看"不加锁会出什么事"。**

假设有个余额扣款函数（先不管什么图谱，就看数字）：

```python
balance = 100                     # alice 的余额

async def spend_10():
    global balance
    current = balance             # ① 读出 100
    await asyncio.sleep(1)        # ② 模拟"要等一下"（比如等数据库、等 LLM）
    balance = current - 10        # ③ 写回 100 - 10 = 90

# 两个请求同时进来：
await asyncio.gather(spend_10(), spend_10())
print(balance)    # 你以为是 80，实际是 90！
```

逐帧回放哪里错了：

```text
请求1：读到 balance = 100          （①）
请求1：await，让出执行权            （②）← 关键：让出去了
请求2：趁机开跑，也读到 balance = 100（①）← 灾难种子：读到的是旧值
请求2：await，让出
请求1：恢复，写回 100 - 10 = 90     （③）
请求2：恢复，写回 100 - 10 = 90     （③）← 把请求1的扣款覆盖了！

两次扣款，只扣掉一次。这就叫【竞态条件】(race condition)：
结果取决于谁先谁后的"巧合"，大部分时候对，偶尔错，极难排查。
```

注意一个精确的认知：**单线程的 asyncio 里，只有在 `await` 处才可能被"插队"**。如果 ①②③ 之间没有任何 await，就不会出事。但你的 `add_episode` 内部有几十个 await（每次调 LLM 都是），插队机会遍地都是。

**加锁之后：**

```python
lock = asyncio.Lock()

async def spend_10():
    global balance
    async with lock:              # 进门先拿钥匙，屋里有人就在门口等
        current = balance
        await asyncio.sleep(1)    # 就算这里让出执行权，
                                  # 别的请求进不了这扇门（拿不到锁）
        balance = current - 10

await asyncio.gather(spend_10(), spend_10())
print(balance)    # 80，永远正确
```

`async with lock:` 的意思：**这段代码同一时刻只允许一个任务在里面**。第二个请求执行到这一行时发现锁被占了，就乖乖排队，等第一个请求出来（释放锁）再进去。

**对应到你的服务器**，`add_episode` 的写入流程本质也是"读-想-写"：

```text
读：查图里有没有"布丁"这个实体节点
想：LLM 判断要不要合并、要不要让旧边失效
写：写入新节点/新边

alice 连发两条消息，如果不加锁并发处理：
两个请求都查到"图里没有布丁" → 各自创建 → 图里出现两个布丁节点
去重逻辑、时序失效逻辑全部被破坏
```

**再讲"per-user"这三个字的价值。** 对比三种方案：

| 方案 | 代码 | 效果 |
|------|------|------|
| 不加锁 | 直接处理 | alice 自己的两个请求互相踩，图数据损坏 |
| 一把全局大锁 | `async with global_lock:` | 数据safe了，但 alice 处理时 **bob 也得等**——全服务器排成一条队，多用户性能归零 |
| per-user 锁（你的方案） | 每个 user_id 一把锁 | alice 的请求之间排队（保正确），alice 和 bob 之间照样并发（保性能） |

一图流：

```text
全局一把锁：   alice#1 → alice#2 → bob#1 → bob#2     （全排队，慢）
per-user 锁：  alice#1 → alice#2                      （alice 队内排队）
               bob#1   → bob#2                        （bob 队内排队，两队并行）
```

最后看懂那行创建锁的代码：

```python
locks.setdefault(user_id, asyncio.Lock())
# 意思：字典里有 alice 的锁就直接返回它；没有就现场造一把存进去再返回。
# 保证同一个 user_id 拿到的永远是【同一把】锁对象——这是锁生效的前提。
# （如果每次都 new 一把新锁，等于每人每次换新门锁，锁了个寂寞）
```

记忆句：**锁保护的不是数据，是"读-想-写"这个过程不被插队；per-user 是把排队范围缩小到刚刚好**。

### 4.5 同一把用户锁会覆盖哪些请求

这把锁不只是让两个 `/memory/add` 排队。当前代码中的 add、delete、graph、search、
response 等用户相关路径都围绕同一个 `user_id` 获取锁，所以可能出现：

```text
alice 正在 ingest（持有 alice 锁）
  → alice 的 search 到达：等待 alice 锁
  → bob 的 search 到达：获取 bob 锁，正常执行
  → /health 到达：不需要用户锁，正常执行
```

这是保守但容易理解的正确性策略：同一用户的读写都串行，避免读到一半状态。代价是一次很慢
的 ingest 会拖住该用户的查询。以后可以把 ingest 放后台队列，或在确认数据库一致性语义后
细分读写锁；不要在没验证 Graphiti 并发写语义前贸然删锁。

## 5. uvicorn workers：它是进程，不是线程

### 5.1 先纠正一个普遍误解

你说的「uvicorn 开多个线程 worker」——**uvicorn 的 worker 是进程（process），不是线程（thread）**。

```bash
uvicorn main_server:app --host 0.0.0.0 --port 8000 --workers 4
```

这条命令启动的东西：

```text
┌── 主进程（只负责管理，不处理请求）──┐
│                                    │
│  ┌─ worker 进程 1 ─┐  各自是完整独立的 Python 解释器
│  │  事件循环 1      │  各自执行一遍 lifespan（各建一个 Graphiti 实例！）
│  │  app.state 1    │  各自有自己的 app.state
│  └─────────────────┘
│  ┌─ worker 进程 2 ─┐
│  │  事件循环 2      │
│  │  app.state 2    │  ← 和 worker 1 的 app.state 【完全无关】
│  └─────────────────┘
│  ┌─ worker 3、4 ...─┘
│
│  操作系统把 8000 端口收到的请求【随机】分给某个 worker
└────────────────────────────────────┘
```

最简单的验证例子（20 行亲眼看"进程不共享内存"）：

```python
# counter_demo.py
from fastapi import FastAPI
import os

app = FastAPI()
counter = 0                        # 全局变量

@app.get('/count')
async def count():
    global counter
    counter += 1
    return {'counter': counter, 'pid': os.getpid()}   # pid = 进程编号
```

```bash
# 单 worker 启动，连续请求 6 次：
# {"counter":1,"pid":1234} {"counter":2,"pid":1234} {"counter":3,"pid":1234} ...
# 计数正常递增，pid 始终是同一个

# 4 worker 启动：uvicorn counter_demo:app --workers 4，连续请求 6 次：
# {"counter":1,"pid":1001} {"counter":1,"pid":1002} {"counter":2,"pid":1001}
# {"counter":1,"pid":1003} {"counter":1,"pid":1004} {"counter":2,"pid":1002}
# 计数乱跳！因为有 4 个进程，每个进程有【自己的】counter，各数各的
```

把这里的 `counter` 换成你的 `user_locks` 字典，就是第 6 节要讲的事故根源。

### 5.2 为什么要开多 worker

1. **增加并行执行 Python 代码的能力**：每个 worker 有独立 GIL，可以利用多个 CPU 核。
   但 4 个 worker 不等于吞吐必然变成 4 倍；I/O 比例、LLM 配额、数据库连接和锁都会成为瓶颈。
2. **容错**：一个 worker 崩了（内存泄漏、段错误），别的 worker 还活着，主进程还会拉起新的。

### 5.3 开几个

经验公式：

```text
CPU 密集为主：workers = CPU 核数
I/O 密集为主：workers = CPU 核数 ～ 核数×2（再多收益递减，asyncio 已经把单进程 I/O 并发榨干了）
```

你的服务是重 I/O 的，单 worker 靠 asyncio 已经能撑很高并发，**开多 worker 主要图容错和多核，不是刚需**。

### 5.4 相关工具澄清

| 名词 | 是什么 | 你需要吗 |
|------|--------|----------|
| uvicorn | ASGI 服务器，跑 async 应用的引擎 | 正在用 |
| `--workers N` | uvicorn 自带的多进程模式 | 学习实验用 |
| gunicorn | 老牌进程管理器，配 uvicorn worker 类使用，**只支持 Linux** | 部署到 Linux 服务器时再说 |
| nginx | 反向代理，做负载均衡/HTTPS/请求边界，架在 uvicorn 前面 | 单机公网部署也常用 |

注意：Windows 上 gunicorn 用不了；uvicorn 的 `--workers` 新版本（0.30+）在 Windows 可用。生产部署基本都在 Linux/Docker 里，别在 Windows 上死磕。

另一个容易混淆的参数：FastAPI 给同步 `def` 接口使用的线程池/线程令牌容量通常是几十量级，
具体值受 Starlette、AnyIO 和你的配置影响；它和 worker 没有任何关系。前者是进程内的线程，
后者是独立进程。

## 6. 开多 worker 后你的代码哪里会炸

这一节是全文档最重要的部分：**你的代码是按单进程假设写的，多进程假设一变，三个地方立刻出问题**。理解为什么会炸，比背 Redis 命令有价值十倍。

### 炸点 1：`asyncio.Lock` 失效

```python
def get_user_lock(request: Request, user_id: str) -> asyncio.Lock:
    locks: dict[str, asyncio.Lock] = request.app.state.user_locks
    return locks.setdefault(user_id, asyncio.Lock())
```

`asyncio.Lock` 是**事件循环内**的锁。4 个 worker = 4 个事件循环 = **同一个 user_id 有 4 把互不知道对方存在的锁**。

事故现场：

```text
用户 alice 连发两个 /memory/add 请求
→ 操作系统把请求1分给 worker1，请求2分给 worker2
→ worker1 拿到自己的 alice 锁 ✓，worker2 也拿到自己的 alice 锁 ✓
→ 两个进程【同时】往 alice 的图里写入
→ Graphiti 的去重和时序失效逻辑假设串行写入，并发写导致重复实体/错误失效
```

解决：锁必须放在所有进程都看得见的地方 → **分布式锁**，最常用的载体就是 Redis（第 7 节）。

### 炸点 2：进度检查与追加不是跨进程原子操作

```python
group_progress = load_register_progress(group_id)            # 读取 JSONL
completed = {item.episode_name for item in group_progress}
if episode_name in completed:
    continue
...
append_register_progress(group_id, progress_item)             # 追加一行 JSONL
```

当前代码已经不是“把整个 JSON 文件读出再覆盖写回”，而是 append-only JSONL，这降低了整文件
覆盖丢数据的风险。但“检查是否完成 → 执行 ingest → 追加完成记录”这三个动作仍不是一个
跨进程原子事务。两个 worker 可能同时判断某条 episode 尚未完成，然后重复 ingest；并发追加
也依赖操作系统文件语义，不能作为多机一致性方案。

单进程时被炸点 1 的锁保护着，没事；多进程时锁先失效，这里跟着炸。

解决：进度状态放 Redis（原子操作 `SADD`）或数据库，别放本地文件。

### 炸点 3：per-user Graphiti 缓存 × N 份

```python
app.state.user_graphiti = {}    # 每个 worker 各有一份
```

4 个 worker 各自缓存一份每个用户的 Graphiti 实例，各自维护数据库连接。内存 ×4，连接数 ×4。不算致命，但要心里有数：**进程内缓存在多进程下天然是"每进程一份"**。

### 缓存补课：它解决什么，又引入什么

缓存就是“把昂贵操作的结果暂存起来，下次先查暂存结果”。它通常换取更快响应，但会引入
**过期、容量和一致性**问题。

`main_server.py` 的 `user_graphiti` 是一个懒加载缓存：

```text
请求 user_id=alice
  → 字典里已有 alice：直接返回 Graphiti 实例（命中）
  → 字典里没有 alice：clone driver、build_indices、存入字典（未命中）
```

这里缓存的不是用户记忆数据，而是 Graphiti 客户端/驱动对象，目的是避免每个请求重复建连接
和索引。缓存设计至少要回答五个问题：

1. **key 是什么**：这里是 `user_id`。
2. **value 是什么**：这里是用户专属 Graphiti 实例。
3. **何时创建**：第一次访问时懒加载。
4. **何时失效**：当前主要靠进程退出；生产中还应考虑用户删除、配置变化、连接失效。
5. **最多存多少**：当前没有上限；用户持续增加时，实例和连接也会持续增加。

常见术语：

- **TTL**：缓存活多久，超时自动失效。
- **LRU**：容量满时，优先淘汰最久没用的条目。
- **缓存穿透**：大量查询不存在的 key，每次都打到后端；可短暂缓存“未找到”。
- **缓存击穿**：某个热点 key 过期，大量请求同时重建；可用 single-flight/锁只让一个请求重建。
- **缓存雪崩**：大量 key 同时过期；TTL 加随机抖动可分散重建时间。

`clear_memory()` 清除的是图数据和进度文件，不等于自动关闭或移除 `user_graphiti` 中的客户端。
是否需要移除，取决于删除语义和驱动状态；至少应把“数据删除”和“缓存失效/连接关闭”作为
两个明确步骤审查，而不是假设缓存会自己更新。

不要把所有缓存都搬进 Redis。客户端实例和数据库连接不能序列化，仍应保留在各进程本地；
适合放 Redis 的是检索结果、用户配置、状态和限流计数等可序列化数据。

### 总结规律

> 单进程时代，「全局变量 + asyncio.Lock」就是你的状态管理。
> 多进程时代，一切需要共享的状态（锁、缓存、进度、会话）都必须搬出进程，
> 搬到一个所有进程都能访问的外部服务里——这个服务通常就是 Redis。

## 7. Redis：进程之间的公共黑板

### 7.1 Redis 到底是什么

一句话：**运行在内存里的键值数据库，命令执行主逻辑以单线程串行为主、速度快、自带丰富
数据结构和原子操作**。现代 Redis 还可能使用 I/O 线程和后台线程，所以“Redis 完全只有一个
线程”并不严谨；这里真正重要的是单条命令的原子语义。

把它想成"所有进程共用的一块公共黑板"：

```text
worker1 ──┐
worker2 ──┼──── Redis（一个独立进程/容器，端口 6379）
worker3 ──┤       key: "lock:alice"      → "worker2持有"
后台任务 ──┘       key: "progress:alice"  → {ep1, ep2, ep3}
                  key: "cache:query:xxx" → "检索结果json"
```

### 7.2 五个核心数据结构（够用版）

| 结构 | 类比 | 典型用途 |
|------|------|----------|
| String | 变量 | 缓存、计数器（`INCR` 原子自增）、分布式锁 |
| Hash | dict | 用户资料、配置 |
| List | 双端队列 | **简易消息队列**（`LPUSH` + `BRPOP`） |
| Set | 集合 | 你的进度文件正确替代品（`SADD`/`SISMEMBER`，原子去重） |
| Sorted Set | 带分数的有序集合 | 排行榜、延迟任务队列 |

外加一个关键特性 **TTL**（过期时间）：`SET key value EX 3600` → 一小时后自动删除。缓存和锁都靠它兜底。

最简单的上手例子（两条命令启动 + 10 行代码跑通）：

```bash
docker run -d --name redis -p 6379:6379 redis:7
pip install redis
```

```python
import asyncio
import redis.asyncio as redis

async def main():
    r = redis.Redis(decode_responses=True)   # 连本机 6379

    await r.set('greeting', '你好')           # 存
    print(await r.get('greeting'))            # 取 → 你好

    await r.set('cache:answer', '42', ex=5)   # 存一个 5 秒后自动消失的 key
    await asyncio.sleep(6)
    print(await r.get('cache:answer'))        # → None（已过期）

    print(await r.incr('visits'))             # 原子计数 → 1
    print(await r.incr('visits'))             # → 2（多进程同时调也不会算错）

asyncio.run(main())
```

体会重点：这段代码不管从哪个进程、哪台机器跑，操作的都是**同一份数据**——这就是"公共黑板"的含义，也是它能修复多 worker 问题的原因。

### 7.3 用 Redis 修你的三个炸点

**修炸点 1 —— 分布式锁**，原理只有一行命令：

```text
SET lock:user:alice <随机值> NX EX 30
```

- `NX` = 不存在才能设置成功 → 谁设置成功谁持锁（原子的，多进程同时抢只有一个赢）
- `EX 30` = 30 秒自动过期 → 持锁进程崩了锁也不会死锁
- 释放锁时校验随机值是自己的才删（防止误删别人的锁）

Python 里不用手写，`redis-py` 自带：

```python
import redis.asyncio as redis

r = redis.Redis()

async def register(...):
    async with r.lock(f'lock:user:{user_id}', timeout=300):
        ...  # 和原来 asyncio.Lock 的代码一样，但现在跨进程有效
```

**修炸点 2 —— 进度用 Set 存**：

```python
# 原来：读 JSONL → 检查 → ingest → 追加完成记录（整体不是跨进程原子操作）
# 现在：
already = await r.sismember(f'progress:{group_id}', episode_name)  # 原子查
...
await r.sadd(f'progress:{group_id}', episode_name)                 # 原子加
```

**修炸点 3 —— 跨进程缓存**：检索结果、用户画像等只读热数据放 Redis 加 TTL，哪个 worker 都能命中。

### 7.4 Redis 还能干什么（先知道名字）

- **限流**：`INCR` + `EX` 实现"每用户每分钟最多 N 次请求"
- **会话存储**：登录态放 Redis，任何 worker 都能验证
- **Pub/Sub**：进程间广播消息
- **Streams**：比 List 更正规的消息队列（带消费者组、确认机制）

启动方式（学习用）：

```bash
docker run -d --name redis -p 6379:6379 redis:7
pip install redis   # Python 客户端，redis.asyncio 支持 async
```

## 8. 消息队列：把慢活从请求里摘出去

### 8.1 你现在的痛点（真实存在）

你的 `/memory/add`：客户端发 100 条消息 → 服务器逐条 `add_episode`（每条几秒，内部多次 LLM 调用）→ **客户端干等几分钟才拿到响应**。等待期间：HTTP 超时风险、客户端不能干别的、服务器重启则全部作废。

### 8.2 消息队列的思想：受理和处理分离

去银行不是站在柜台前盯着办完，而是**取号 → 走人 → 叫号**。

```text
──────────── 改造前（同步处理）────────────
客户端 → POST /memory/add → [处理3分钟] → 返回结果
         （客户端全程挂着等）

──────────── 改造后（异步处理）────────────
客户端 → POST /memory/add → 任务塞进队列 → 立刻返回 {"task_id": "abc"}   ← 毫秒级
                                 ↓
                       后台 worker 从队列取任务，慢慢处理
                                 ↓
客户端 → GET /tasks/abc → {"status": "processing", "done": 40, "total": 100}
客户端 → GET /tasks/abc → {"status": "completed"}
```

### 8.3 消息队列的三大作用

| 作用 | 说明 | 你的场景 |
|------|------|----------|
| 异步化 | 请求立刻返回，慢活后台干 | add_episode 摘出请求路径 |
| 削峰 | 瞬间 1000 个请求先排队，后端按自己的速度消化 | 评测脚本突发灌入大量消息时不压垮 LLM 配额 |
| 解耦 | 生产者和消费者互不认识，可独立扩容/重启 | API 服务器和 ingest worker 分开部署、分开扩容 |

### 8.4 实现的三个档位

**档位 1：进程内 asyncio 后台任务（零新依赖，单 worker 限定）**

```python
# lifespan 里启动常驻消费者
queue: asyncio.Queue = asyncio.Queue()

async def ingest_worker():
    while True:
        task = await queue.get()
        try:
            await do_ingest(task)
        finally:
            queue.task_done()

# 接口里只入队
@app.post('/memory/add')
async def register(request: RegisterRequest):
    await queue.put(request)
    return {'task_id': ..., 'status': 'queued'}
```

- 优点：50 行代码搞定，立刻能用。
- 缺点：任务在内存里，**进程重启任务全丢**；多 worker 下每个进程一条队列，没法全局排队。
- 适合：你的当前阶段。配合你已有的进度持久化，重启后客户端重发即可续传（幂等）。

**档位 2：Redis 作队列 + 独立 worker 进程（生产小规模主流）**

用现成库，别手写。async 生态推荐 **ARQ**（基于 redis，极简），或者老牌 **Celery**（功能全但重，async 支持一般）：

```python
# ARQ 示例：定义任务
async def ingest_messages(ctx, user_id: str, messages: list[dict]):
    ...  # add_episode 逻辑

# API 进程里投递任务
job = await arq_pool.enqueue_job('ingest_messages', user_id, messages)

# 单独的命令行启动 worker 进程（和 API 进程完全分开）
# arq worker_settings
```

- 任务持久化在 Redis，进程崩了任务还在，worker 重启继续消费。
- API 进程和 worker 进程可以在不同机器上，各自扩容。
- 这一档具备小规模生产任务队列的基础，但还必须补齐幂等、超时、重试、取消、监控和容量上限。

**档位 3：专业消息队列（RabbitMQ / Kafka，知道概念即可）**

| | RabbitMQ | Kafka |
|---|---|---|
| 定位 | 任务分发（路由灵活、确认机制完善） | 日志流（超高吞吐、消息可回放） |
| 典型场景 | 订单处理、任务调度 | 用户行为流、日志管道、大数据 |
| 你需要吗 | Redis 队列扛不住时再学 | 做数据管道时再学 |

面试标准话术：「小规模用 Redis + ARQ/Celery 足够；需要复杂路由和可靠投递上 RabbitMQ；超高吞吐的流式数据上 Kafka。」

### 8.5 用了队列后必须处理的两件事

1. **幂等**：任务可能被重复执行（worker 崩溃后任务重投）。你的进度记录（改造成 Redis Set 后）就是幂等保障——重复执行时已 ingest 的消息自动跳过。你现在的设计思想已经对了。
2. **任务状态可查**：给每个任务发 `task_id`，状态（queued/processing/completed/failed + 进度）写 Redis，提供 `GET /tasks/{id}` 接口。

## 9. 用户隔离

### 9.1 隔离的四个层级（从松到紧）

| 层级 | 做法 | 成本 | 你的现状 |
|------|------|------|----------|
| L1 逻辑隔离 | 同一张表/图，数据带 user_id 字段，查询时过滤 | 最低 | ✓ 你在用（group_id） |
| L2 库级隔离 | 每个用户一个 database/schema | 中 | ✓ 你在用（driver.clone(database=user_id)） |
| L3 实例隔离 | 每个用户/租户单独一套服务进程+数据库 | 高 | 无需 |
| L4 物理隔离 | 单独的机器/集群（金融、医疗合规场景） | 极高 | 无需 |

你的代码同时用了 L1 和 L2：

```python
# L2：每个用户克隆一个指向独立 database 的 driver
driver = base_graphiti.driver.clone(database=user_id)

# L1：写入和查询都带 group_id
await graphiti.add_episode(..., group_id=group_id)
await graphiti.search_(query, group_ids=[group_id])
```

**评价：对单机多用户场景，这个设计已经超出"简单接口部署"的水平了**。双保险（库隔离 + group_id 过滤）意味着即使一层出 bug，另一层还兜着，用户 A 的记忆不会泄漏给用户 B。

### 9.1.1 用最简单的例子讲清"加和不加"的区别

这段代码也是 AI 帮你写的，现在拆开看每一层到底防住了什么。

**先看什么都不加的裸奔版。** 想象所有用户的记忆存在同一张表里，没有任何用户标记：

```text
记忆表（所有人共用）：
┌──────────────────────────┐
│ 我的猫叫布丁               │   ← alice 说的
│ 我在字节上班               │   ← alice 说的
│ 我对花生过敏               │   ← bob 说的
└──────────────────────────┘

bob 问："我的宠物叫什么？"
系统检索 → 命中"我的猫叫布丁" → AI 回答 bob："你的猫叫布丁"
```

**这就是不隔离的后果：bob 看到了 alice 的私事**。对陪伴产品这是灾难级事故（想象用户 A 的感情秘密被说给用户 B 听）。

**第一层：L1 逻辑隔离（group_id）= 给每条数据贴名字标签。**

```text
记忆表（还是同一张表，但每行多了一个标签列）：
┌──────────────────────┬──────────┐
│ 我的猫叫布丁           │ alice    │
│ 我在字节上班           │ alice    │
│ 我对花生过敏           │ bob      │
└──────────────────────┴──────────┘

bob 问"我的宠物叫什么？"
检索时多带一个条件：只在标签 = bob 的行里找
→ 找不到布丁 → 正确
```

对应你代码里的：

```python
await graphiti.add_episode(..., group_id=group_id)      # 写入时贴标签
await graphiti.search_(query, group_ids=[group_id])     # 查询时按标签过滤
```

L1 的优点：便宜，一个字段搞定。
L1 的弱点：**它依赖每一处查询都记得带过滤条件**。系统里有 100 个查询点，99 个带了 group_id，漏掉 1 个，数据就泄漏。这种 bug 还特别隐蔽——功能一切正常，只是"偶尔多查出别人的东西"。

**第二层：L2 库级隔离（database=user_id）= 每人一个独立房间。**

```text
不再是一张大表贴标签，而是：

alice 的数据库          bob 的数据库
┌──────────────┐      ┌──────────────┐
│ 我的猫叫布丁    │      │ 我对花生过敏   │
│ 我在字节上班    │      └──────────────┘
└──────────────┘

处理 bob 的请求时，连接的【物理上就是 bob 的库】。
就算某处查询忘了带 group_id 过滤——也查不到 alice 的数据，
因为 alice 的数据根本不在这个库里。
```

对应你代码里的：

```python
driver = base_graphiti.driver.clone(database=user_id)
# 意思：复制一个数据库连接，但指向名为 <user_id> 的独立 database
# 之后这个用户的所有读写，都发生在他自己的库里
```

**为什么两层都要（双保险的真实含义）：**

| 场景 | 只有 L1 | 只有 L2 | L1 + L2（你的方案） |
|------|---------|---------|---------------------|
| 某处查询代码忘带 group_id | **泄漏** | 安全（库里本来就没别人的数据） | 安全 |
| 连接路由出 bug，把 bob 的请求连到了 alice 的库 | 安全（group_id 过滤兜底） | **泄漏** | 安全 |
| 两个 bug 同时发生 | 泄漏 | 泄漏 | 泄漏（概率极低） |

一句话：**L1 是"查询时过滤"，防的是连接错库；L2 是"物理分家"，防的是查询漏过滤。单层靠"代码处处不出错"，双层靠"两层同时出错才出事"**——后者才是工程上可靠的安全。

顺带一个成本认知：L2 不是免费的。每个用户一个 database 意味着索引、连接各一份，用户到万级时管理成本上来了——所以生产上常见的折中是"大客户单独建库（L2），海量小用户共库靠字段过滤（L1）"。你现在两层都上是评测场景（用户少）的奢侈打法，完全合理。

### 9.2 你缺的一块：认证（Authentication）

现在任何人只要知道你的地址，就能传任意 `user_id` 读写任何人的记忆——隔离做了，**门没锁**。

最小改造（API Key 方案，30 行）：

```python
from fastapi import Security, HTTPException
from fastapi.security import APIKeyHeader

api_key_header = APIKeyHeader(name='X-API-Key')

# key → user_id 的映射（正式做法存数据库/Redis，学习阶段字典即可）
API_KEYS = {'sk-alice-xxx': 'alice', 'sk-bob-yyy': 'bob'}

async def get_current_user(api_key: str = Security(api_key_header)) -> str:
    if api_key not in API_KEYS:
        raise HTTPException(status_code=401, detail='Invalid API key')
    return API_KEYS[api_key]

@app.post('/memory/add')
async def register(
    request: RegisterRequest,
    user_id: str = Depends(get_current_user),   # ← user_id 由服务器根据 key 得出
):
    ...  # 不再信任请求体里的 user_id
```

核心原则一句话：

> **身份永远由服务端从凭证推导，永远不信任客户端自己报的 user_id。**

再往上是 JWT（token 里带签名的用户信息，适合有登录流程的产品）、OAuth2（第三方登录），概念先放着，需要时再学。

### 9.3 「隔离环境」的另一个含义：资源隔离

除了数据隔离，多用户还要防止一个用户吃光资源饿死其他人：

- **限流**：每用户每分钟最多 N 次请求（Redis `INCR` + TTL，10 行代码）
- **配额**：每用户每天 LLM token 上限（你的 `RequestUsage` 统计已经把地基打好了，加个累计判断就是配额系统）
- **队列公平**：任务队列按用户轮转消费，防止一个用户灌 1 万条消息堵死队列

## 12. 最终记忆版

并发模型：

```text
并发 = 交替，并行 = 同时
I/O 密集 → asyncio（等待时让出；容量由连接、内存和下游上限共同决定）
纯 Python CPU 密集 → 多进程/原生扩展（普通线程通常受 GIL 限制）
await = "让"，不是"等"：让出执行权给别的任务，完成后从原地继续
async def 里写同步阻塞 = 卡死整个事件循环 = 最经典事故
```

锁：

```text
竞态条件 = 读-想-写的过程被人插队（asyncio 里只在 await 处可能被插队）
锁 = 同一时刻只许一个任务进入这段代码
per-user 锁 = 排队范围缩到刚刚好：同一用户排队保正确，不同用户并发保性能
锁必须是"同一把"才有效（setdefault 保证），多进程下要换 Redis 分布式锁
```

uvicorn worker：

```text
worker = 独立进程，不是线程
每个 worker：独立解释器 + 独立事件循环 + 独立 app.state
进程不共享内存 → asyncio.Lock/全局 dict 只能管本进程；本地进度文件不能保证跨进程一致性
```

Redis：

```text
定位 = 所有进程共用的内存黑板
四大用途：分布式锁(SET NX EX) / 共享缓存(TTL) / 队列(List/Streams) / 计数限流(INCR)
一切"需要跨进程共享的状态"都搬进它
```

消息队列：

```text
思想 = 受理和处理分离（取号走人，别站柜台等）
作用 = 异步化 + 削峰 + 解耦
档位 = asyncio.Queue(单机) → Redis+ARQ(标准答案) → RabbitMQ/Kafka(知道即可)
配套 = 幂等(防重复执行) + task_id(状态可查)
```

用户隔离：

```text
不隔离 = bob 能检索到 alice 的记忆（陪伴产品的灾难级事故）
L1 字段过滤(group_id) = 数据贴标签，查询时过滤 → 防"连错库"，怕"漏过滤"
L2 库隔离(database per user) = 物理分家 → 防"漏过滤"，怕"连错库"
两层同时上 = 两层同时出错才泄漏 ← 你已做到
身份认证：user_id 必须由服务端从 API Key/JWT 推导，永不信客户端自报
资源隔离：限流 + 配额 + 队列公平
```

最重要的一句：

> 单进程时可重建的临时状态可以放进程内；多进程必须共享且保持一致的状态放 Redis/数据库。
> 你的代码从“单进程正确”升级到“多进程核心状态一致”，重点要改锁、进度、任务状态/队列；
> 从“多进程能跑”升级到“生产可用”，还必须补认证、边界、限流、超时、可观测和故障测试。

# Part B：生产级 FastAPI 接口

前 12 节主要回答“并发和部署为什么这样设计”。这一部分回答“接口公开给真实用户之前，还要
挡住哪些风险”。生产级不是某个框架开关，而是一组可以验证的约束：

- 谁能调用，能操作谁的数据；
- 一次最多交多少数据，一段时间最多调用多少次；
- 系统最多同时处理多少慢任务，满载后如何拒绝或排队；
- 调用超时、断线、取消或重试后，数据是否仍然正确；
- 出问题时能否快速发现、定位和安全恢复。

下面的代码多数是教学骨架，不能复制几行就宣称“生产安全”。多 worker 场景下，共享状态必须
放 Redis/数据库，管理员接口还必须有独立的管理员认证和审计日志。

## 13. 生产请求的一生：先建立全景图

一个请求从公网进入到完成，通常经过这些关卡：

```text
客户端
  → CDN/WAF/反向代理：TLS、IP 限制、请求体大小、连接超时
  → Uvicorn/FastAPI：解析 HTTP、生成 request_id
  → 认证：这个人是谁
  → 授权：他能否操作这个 user/tenant
  → 输入验证：字段长度、消息条数、格式是否合法
  → 限流/配额：这一分钟和今天是否还能用
  → 并发控制：系统是否还有处理槽位
  → 业务锁：同一用户的数据是否需要串行
  → 队列或业务执行：调用 LLM、数据库
  → 统一响应/错误映射
  → 日志、指标、追踪
```

每层解决的问题不同，不能相互替代：

| 机制 | 回答的问题 | 不能解决什么 |
|------|------------|--------------|
| 认证 | 调用者是谁 | 他是否调用过多 |
| 授权 | 他能做什么 | 系统是否过载 |
| 输入上限 | 单次请求有多大 | 单位时间请求次数 |
| 限流 | 单位时间允许多少请求 | 已经受理的任务如何调度 |
| Semaphore | 同时执行多少个 | 同一用户写图的竞态 |
| per-user 锁 | 同一用户是否串行 | 恶意用户无限排队 |
| 队列 | 满载时暂存、削峰 | 无限容量队列仍会撑爆内存 |
| 超时 | 最长愿意等多久 | 已提交数据自动回滚 |

### 13.1 当前 `main_server.py` 处在哪

当前已经有：

- Pydantic 请求/响应模型和一部分字段约束；
- 单 worker 下的 per-user 锁；
- 每用户 database + `group_id` 数据隔离；
- episode 名称和 JSONL 进度带来的基础幂等；
- lifespan 关闭资源、文本日志、简单 `/health`；
- 请求级 Token 用量统计。

当前还没有：

- 真实认证与操作级授权；
- 每用户/IP 限流、每日配额执行；
- `messages` 等批量数量和 HTTP 请求体字节上限；
- 全局并发上限、有限容量队列和过载拒绝；
- 管理员封禁/取消用户、任务状态接口；
- 系统性的超时、断连处理和重试策略；
- readiness、指标、分布式追踪、统一错误结构；
- 面向公网的 HTTPS、反向代理和安全加固。

所以它是“功能能跑、评测够用”，不是“可以不加防护直接暴露公网”。

## 14. 认证、授权与管理员停用用户

### 14.1 认证和授权不是一回事

- **认证（Authentication）**：确认“你是谁”，例如 API Key/JWT。
- **授权（Authorization）**：确认“你能做什么”，例如普通用户不能清除别人的记忆。

当前请求体里的 `user_id` 由客户端填写。攻击者把它改成 `alice`，服务端就会以 alice 身份
读写数据。正确流程是：

```text
X-API-Key/JWT
  → 服务端验证签名或查 key
  → 得到可信 principal(user_id, tenant_id, roles)
  → 路由只使用 principal.user_id
  → 忽略或拒绝请求体中自报的 user_id
```

API Key 适合机器调用；JWT 适合有登录和过期机制的用户系统；OAuth2 是授权协议，不等于
“用了 OAuth2 就自动安全”。无论哪种方式，都要考虑过期、撤销、轮换和泄漏后的处置。

授权至少区分：

- `memory:read`：search、response、graph；
- `memory:write`：add；
- `memory:delete`：delete；
- `admin:user_control`：封禁、解封、取消任务。

### 14.2 服务端停止某个用户的正确模型

一个用户不是一个进程，因此不能通过“杀掉用户进程”解决。应组合三个动作：

1. **封禁**：把 `user_id` 写入 Redis/数据库，所有新请求入口检查后返回 `403`。
2. **撤销排队任务**：把 queued 任务标为 cancelled，worker 取到后不执行。
3. **取消运行任务**：通知每个 worker 取消本进程里属于该用户的 Task。

单 worker 的教学骨架：

```python
blocked_users: set[str] = set()
running_tasks: dict[str, set[asyncio.Task]] = {}

async def reject_blocked(user_id: str) -> None:
    if user_id in blocked_users:
        raise HTTPException(status_code=403, detail='User disabled')

async def run_for_user(user_id: str, operation):
    task = asyncio.current_task()
    if task is None:
        raise RuntimeError('No current task')

    running_tasks.setdefault(user_id, set()).add(task)
    try:
        await operation()
    finally:
        running_tasks[user_id].discard(task)

async def stop_user_locally(user_id: str) -> None:
    blocked_users.add(user_id)
    for task in tuple(running_tasks.get(user_id, ())):
        task.cancel()
```

生产多 worker 版本不能只用上述全局字典：

```text
管理员请求
  → 数据库/Redis 写 disabled:user_id
  → Redis Pub/Sub 广播 cancel:user_id
  → 每个 API/worker 进程收到广播
  → 取消自己进程内该用户 Task
  → 每个接口入口和队列 worker 都检查 disabled 状态
```

必须注意：

- 管理员接口要有强认证、最小权限、操作原因和审计日志。
- `task.cancel()` 是协作式取消，不是操作系统强杀。
- 已发到 LLM 的请求可能仍在供应商侧执行并计费。
- 已提交到数据库的数据不会因为 Python Task 取消而自动删除。
- Graphiti 的一次 `add_episode()` 可能含多次 LLM/数据库步骤，半途取消可能留下部分结果。

因此更安全的策略通常是：**立即禁止后续消息，在当前 episode 完成后的安全停止点退出**；
只有在成本失控或安全事件时才尝试硬取消当前 await。

### 14.3 API 返回码

| 场景 | 建议状态码 |
|------|------------|
| 没提供或凭证无效 | `401 Unauthorized` |
| 身份有效但无权限/用户被禁用 | `403 Forbidden` |
| 用户不存在且允许暴露这个事实 | `404 Not Found` |
| 管理操作已受理、取消仍在传播 | `202 Accepted` |

## 15. 输入边界、限流、配额与并发上限

“限制请求”至少有四个维度。只做每分钟 60 次仍挡不住“一次提交一百万条消息”。

### 15.1 单次请求边界

应限制：

- HTTP body 总字节数，例如 1 MiB 或按业务测算；
- `messages`、`queries`、`qa` 的元素数量；
- 每条 `content`、`speaker`、`user_id` 的字符长度；
- 图查询的 `limit`；
- JSON 嵌套深度、文件类型和上传大小（如果以后支持上传）。

Pydantic 负责解析后的字段边界：

```python
from typing import Annotated
from pydantic import BaseModel, Field

class Message(BaseModel):
    speaker: Annotated[str, Field(min_length=1, max_length=100)]
    content: Annotated[str, Field(min_length=1, max_length=20_000)]

class RegisterRequest(BaseModel):
    messages: Annotated[list[Message], Field(min_length=1, max_length=100)]
```

HTTP body 字节上限更适合在 nginx/API Gateway/WAF 先挡一层，因为等 FastAPI 把巨大 body 全部
读进内存后才验证已经太晚。应用层仍应保留字段限制作为第二层。

状态码区分：

- `413 Payload Too Large`：HTTP body 太大；
- `422 Unprocessable Entity`：字段格式、长度、元素数量不合法；
- `429 Too Many Requests`：频率或配额超限。

### 15.2 限流的三个主体

| 限制主体 | 作用 | 注意 |
|----------|------|------|
| IP | 认证前阻挡扫描和暴力请求 | NAT 下很多用户可能共用 IP |
| user/API key | 防止单个已认证用户滥用 | 必须先获得可信身份 |
| 全局 | 保护整个服务和下游配额 | 满载时所有人都可能被拒绝 |

常见算法：

- **固定窗口**：每分钟一个计数器，最简单；窗口交界可能瞬间放过两倍流量。
- **滑动窗口**：统计最近 60 秒，更平滑；Redis Sorted Set 成本更高。
- **令牌桶**：按速度补充令牌，允许有限突发；适合 API。
- **漏桶**：按固定速度排出，输出最平滑；常用于整形。

Redis 固定窗口教学示意：

```python
async def allow_request(redis, user_id: str, limit: int = 60) -> bool:
    minute = int(time.time() // 60)
    key = f'rate:{user_id}:{minute}'
    count = await redis.incr(key)
    if count == 1:
        await redis.expire(key, 70)
    return count <= limit
```

生产中计数和设置过期最好放 Lua 脚本保证原子性；响应 `429` 时返回 `Retry-After`，告诉客户端
多久后重试。限流 key 必须有 TTL，防止 Redis 永久积累。

### 15.3 配额与限流不同

- 限流：控制短时间速度，例如 60 请求/分钟。
- 配额：控制长期总量，例如 100 万 Token/天、1 万条消息/月。
- 并发上限：控制此刻同时运行多少个，例如最多 20 个 LLM 调用。

当前 `RequestUsage` 只是统计本次 Token，没有累计和拒绝逻辑。生产配额要在 Redis/计费数据库
累计，并考虑失败调用是否计费、跨日时区、供应商账单对账和并发扣减的原子性。

### 15.4 四种“排队工具”不要混用

```text
每用户限流      → 防止 alice 高频提交
全局 Semaphore  → 系统同时最多执行 N 个昂贵调用
per-user Lock   → alice 同一时间只做一个会破坏一致性的图操作
有限容量 Queue  → 承接短时突发，满了就拒绝
```

全局并发槽位示意：

```python
llm_slots = asyncio.Semaphore(20)

async def guarded_add_episode(...):
    try:
        async with asyncio.timeout(5):       # 最多等槽位 5 秒
            async with llm_slots:
                return await graphiti.add_episode(...)
    except TimeoutError:
        raise HTTPException(
            status_code=503,
            detail='Server busy',
            headers={'Retry-After': '5'},
        )
```

`Semaphore` 只限制单进程；多 worker 的全局上限需要 Redis 信号量、队列 worker 数量或网关级
限制。队列也必须设置 `maxsize`，无限队列只是把“服务立即崩”变成“内存慢慢涨到崩”。

### 15.5 公平调度

一个 FIFO 总队列可能被 alice 的 10,000 个任务占满，让 bob 永远排在后面。常见办法是：

- 每用户队列 + 用户轮转（round-robin）；
- 每用户设置最大 queued 数量；
- 付费等级使用加权公平队列；
- 单个大批次拆成小 chunk，让其他用户有插队机会。

公平不等于每人完全相同，而是任何单一租户都不能无限占用共享资源。

## 16. 超时、取消、断连、重试与幂等

### 16.1 超时要分层

| 超时 | 控制什么 |
|------|----------|
| 客户端 timeout | 客户端最多等多久；不保证服务端停止 |
| 反向代理 timeout | 连接在网关最多占多久 |
| 接口总超时 | 整个业务请求最多执行多久 |
| LLM/HTTP read timeout | 一次外部请求多久无响应算失败 |
| DB query timeout | 单条数据库查询最多执行多久 |
| 队列任务 timeout | 一个后台任务最长运行多久 |

外层超时不能替代内层超时。若接口总超时 60 秒，但一次 LLM 调用没有 timeout，它可能一直占用
连接直到外层取消；底层库还可能吞掉取消或继续远端执行。

Python 3.11+ 示例：

```python
try:
    async with asyncio.timeout(120):
        await graphiti.add_episode(...)
except TimeoutError:
    # 记录 timeout 类型、user_id、task_id；不要把堆栈直接返回客户端
    raise HTTPException(status_code=504, detail='Upstream operation timed out')
```

### 16.2 取消是“提出请求”，不是强制拔电

`task.cancel()` 会安排在协程下一次可取消点抛出 `asyncio.CancelledError`：

```python
try:
    await graphiti.add_episode(...)
except asyncio.CancelledError:
    logger.info('request cancelled', extra={'user_id': user_id})
    raise                             # 必须继续向上传播
finally:
    # 释放本地登记、临时文件等；async with 会负责释放锁
    unregister_running_task(...)
```

不要写 `except Exception: pass` 吞掉取消。不同 Python 版本中 `CancelledError` 的继承关系有变化，
最稳妥的方式是显式捕获、清理后重新 `raise`。

远端调用的真实效果：

- **LLM**：本地 HTTP 连接会尝试关闭，但模型可能已开始推理，仍可能计费。
- **数据库查询**：驱动可能取消或关闭连接；数据库服务端查询未必立即停止。
- **已提交事务**：不会因 Task 取消自动反向撤销。
- **流式响应**：停止读取和关闭流通常更快，但仍取决于供应商。

对批量 ingest，推荐只在 episode 边界检查取消：

```python
for message in messages:
    if await is_cancelled(task_id):
        await mark_task_cancelled(task_id)
        break

    await graphiti.add_episode(...)   # 尽量完整处理当前 episode
    await mark_episode_completed(...)
```

### 16.3 客户端断开不等于业务自动停止

浏览器关闭、客户端 timeout、网络断开后，FastAPI 路由可能仍继续执行。长同步请求可在安全点检查：

```python
if await request.is_disconnected():
    # 只在业务允许时停止；若任务已正式受理，通常应转后台继续
    raise asyncio.CancelledError
```

对“提交后必须完成”的任务，正确做法是返回 `202 + task_id`，由后台队列继续，而不是把任务生命
绑在 HTTP 连接上。对“用户离开就没有价值”的生成任务，断线后取消更合理。

### 16.4 重试必须和幂等一起设计

只对**瞬时、可恢复**错误重试，例如网络抖动、429、部分 503；不要对认证失败、参数错误、
内容过滤等永久错误反复重试。

指数退避示意：

```text
第 1 次失败 → 等 1s + 随机抖动
第 2 次失败 → 等 2s + 随机抖动
第 3 次失败 → 等 4s + 随机抖动
达到上限 → 进入 failed/dead-letter，等待人工或定时处理
```

随机抖动可避免大量任务同一秒一起重试形成“惊群”。

如果写入操作重试后会重复创建数据，就必须先有幂等键：

- 当前 episode 名称可作为内容级幂等标识，但要确认名称生成不会碰撞；
- HTTP 层可接收 `Idempotency-Key`，保存第一次请求的状态和结果；
- 数据库应有唯一约束或原子 `SETNX`/事务，不能只靠“先查再写”；
- 幂等记录要有合理 TTL，并定义相同 key 但不同请求体如何处理（通常返回冲突）。

记住：**at-least-once 队列一定可能重复投递；幂等不是优化，是正确性前提。**

## 17. 连接池、缓存、队列与背压

### 17.1 连接池为什么需要限制

建立 TCP/TLS/数据库连接很贵，连接池会复用已有连接。但连接池不是越大越好：

```text
总连接数 ≈ API worker 数
         × 每个 worker 的用户客户端实例数
         × 每实例可能持有的连接数
       + 后台 worker 连接
```

如果每个 user 都缓存一个 Graphiti/driver，用户数和 worker 数相乘后可能迅速打满 FalkorDB。
生产前应确认并压测：

- HTTP/LLM 客户端的 `max_connections`、keep-alive 和 read/connect timeout；
- 数据库池最小/最大连接、空闲回收、获取连接 timeout；
- `driver.clone(database=user_id)` 是否共享底层池；
- `user_graphiti` 是否需要 LRU/TTL、淘汰时如何 `close()`；
- 下游允许的最大并发和供应商 RPM/TPM。

连接池满时应有限等待并超时，而不是无限挂起。连接池指标至少包括“正在使用、空闲、等待者、
获取超时”。

### 17.2 本地缓存和 Redis 缓存的边界

| 数据 | 更适合哪里 | 理由 |
|------|------------|------|
| Graphiti/HTTP 客户端实例 | worker 本地 | 对象不可序列化，绑定连接池/事件循环 |
| 热查询结果 | Redis 或本地短 TTL | 可序列化、可跨 worker 共享 |
| 用户禁用状态 | Redis/数据库 | 所有 worker 必须一致看到 |
| 最终任务状态 | Redis/数据库 | 进程重启后不能丢 |
| 极短期请求内数据 | 普通局部变量/ContextVar | 只属于一个请求 |

缓存结果必须把 tenant/user、查询参数、模型/版本等影响结果的字段都放进 key，否则可能发生
跨用户串数据。敏感缓存还要设置 TTL、访问控制和删除策略。

### 17.3 队列不是无限缓冲区

任务状态建议采用明确状态机：

```text
queued → processing → completed
                   ├→ failed
                   ├→ retrying → processing
                   └→ cancelled
```

至少保存 `task_id`、可信 `user_id`、状态、创建/开始/完成时间、进度、重试次数和脱敏后的错误码。
需要定义：

- 最大队列长度、每用户最大排队数；
- worker 崩溃后任务何时重新可见（visibility timeout）；
- 最大重试次数和死信队列；
- 任务状态保留多久；
- 取消 queued 和 processing 任务分别怎么处理；
- 同一用户任务是否必须保持顺序。

### 17.4 背压：下游吃不下时，上游必须慢下来

没有背压时，请求进入速度大于处理速度，队列和内存会持续增长，最后全服务崩溃。背压手段：

1. 提前 `429` 限制滥用用户；
2. 全局并发槽位限制昂贵调用；
3. 有界队列吸收短突发；
4. 队列满时返回 `503 + Retry-After`，不要无限等待；
5. 根据 queue depth、延迟和下游 429 自动降低受理速度；
6. 客户端使用指数退避，不能立即死循环重试。

消息队列解决“什么时候处理”，Semaphore 解决“同时处理几个”，限流解决“允许谁以多快提交”。
生产系统通常三者都需要。

## 18. 错误处理与可观测性

### 18.1 对外错误和内部异常分离

不要把 Python 堆栈、数据库地址、API key 或供应商原始响应直接返回客户端。统一错误结构例如：

```json
{
  "error": {
    "code": "UPSTREAM_TIMEOUT",
    "message": "Memory processing timed out",
    "request_id": "req_abc123",
    "retryable": true
  }
}
```

内部日志保留异常类型和堆栈，对外只给稳定 `code`、安全描述、`request_id` 和是否可重试。

常见映射：

| 情况 | 状态码 | 是否通常可重试 |
|------|--------|----------------|
| 输入不合法 | 422 | 否，先改请求 |
| 未认证/无权限 | 401/403 | 否，先更新凭证或权限 |
| 资源不存在 | 404 | 否 |
| 幂等键冲突 | 409 | 否，检查请求 |
| 请求过大 | 413 | 否，拆批 |
| 限流 | 429 | 是，遵循 `Retry-After` |
| 服务过载 | 503 | 是，退避 |
| 上游超时 | 504 | 视幂等性决定 |

FastAPI 可通过全局 `exception_handler` 保证响应结构一致，但不要把所有异常都变成 `200`；
状态码是客户端、网关和监控判断结果的重要信号。

### 18.2 request ID 和结构化日志

每个请求入口生成或验证 `X-Request-ID`，并贯穿：

```text
API access log
  → Graphiti 调用日志
  → LLM 请求 metadata（供应商支持时）
  → 数据库查询/后台 task_id
  → 错误响应
```

日志建议使用 JSON 字段，而不是把所有内容拼成一句话：

```text
timestamp, level, request_id, task_id, user_id_hash,
route, method, status_code, duration_ms,
queue_wait_ms, llm_duration_ms, db_duration_ms,
prompt_tokens, completion_tokens, error_code
```

不要记录 API key、JWT、完整用户消息或未脱敏个人信息。`user_id` 可按排障需要做哈希或受控记录。

`ContextVar` 适合保存“当前请求的 request_id/用量统计”：每个 Task 有独立上下文，在深层函数
不必层层传参。但创建后台 Task 时要明确上下文是否应该复制，结束时必须 reset，不能把它当
跨进程共享存储。

### 18.3 指标、日志、追踪分别回答什么

- **日志（logs）**：某一次具体发生了什么。
- **指标（metrics）**：整体是否异常、趋势如何。
- **追踪（traces）**：一次请求跨 API、LLM、DB、队列分别花了多久。

生产最小指标：

- 请求数，按 route/status 分类；
- 延迟 p50/p95/p99；
- 当前 in-flight 请求/LLM 调用；
- 429/503/504 比例；
- 队列长度、最老任务等待时间、失败/重试数；
- LLM Token、费用估算、供应商 429；
- 数据库连接池使用率、查询延迟；
- worker 重启次数和事件循环延迟。

告警应对应用户影响，例如“5 分钟内 5xx > 2%”“队列最老任务 > 10 分钟”，而不是仅仅
“CPU > 80%”。Graphiti 已有 OpenTelemetry 相关模块，可在真正接入时统一 trace context；
文档提到能力不代表当前示例服务已配置采集端。

### 18.4 测试不能只验证正常路径

至少覆盖：

- 认证：无 key 为 401，普通用户不能调用 admin；
- 输入边界：空列表、超长 content、101 条消息、超大 body；
- 限流：第 N+1 次返回 429，带 `Retry-After`；
- 幂等：同 key 重试不会重复写入；
- 并发：同用户串行，不同用户可并发；
- 取消：queued/processing 状态都能进入 cancelled；
- 异常：LLM 429/timeout、数据库断连、Redis 断连；
- 生命周期：SIGTERM 时不接新任务并正确关闭连接；
- 隔离：任何查询都不能返回另一个 tenant 的数据。

测试层次：

```text
单元测试       → Pydantic、限流算法、状态机
API 集成测试   → FastAPI + TestClient/AsyncClient
依赖集成测试   → 真实测试 Redis/FalkorDB
契约测试       → OpenAPI 与客户端兼容
压测           → Locust/k6 测 p95、过载和恢复
故障演练       → kill worker、断 Redis、让 LLM 超时
```

## 19. 健康检查、安全部署与优雅停机

### 19.1 liveness、readiness、startup 不能混为一个 `/health`

| 探针 | 问题 | 失败后的动作 |
|------|------|--------------|
| startup | 应用是否完成慢启动/建索引 | 继续等待，不急着重启 |
| liveness | 进程/事件循环是否还活着 | 持续失败才重启容器 |
| readiness | 现在是否适合接新流量 | 从负载均衡摘除，不一定重启 |

当前 `/health` 永远返回 `ok`，只能算最简 liveness。readiness 可检查数据库连接和关键内部状态，
但不要每秒真实调用一次收费 LLM；可以检查客户端配置、近期调用状态或低频探测。依赖短暂异常时
让 readiness 失败比盲目重启容器更合理。

### 19.2 反向代理和应用各自负责什么

nginx/Caddy/API Gateway 常负责：

- TLS/HTTPS 终止、证书续期；
- HTTP body 字节上限、连接/header 超时；
- 基础 IP 限流、WAF、访问日志；
- 多 API 实例负载均衡；
- 设置可信代理头。

FastAPI 仍负责：

- 认证后的 user 级限流和业务配额；
- 字段级验证和租户授权；
- 业务并发、队列、幂等和错误语义。

不能因为前面有 nginx 就删除应用层校验；也不要盲目信任客户端提供的
`X-Forwarded-For`。只有来自可信代理的转发头才能用于识别来源 IP。

### 19.3 CORS、HTTPS 和安全头

- **CORS** 是浏览器跨域读取策略，不是服务器身份认证。只允许明确的前端 origin、method、
  header；带凭证时不能随意用 `*`。
- **HTTPS** 防止 API key/JWT 和用户数据在传输中被窃听或篡改；公网服务必须使用。
- 常见安全头包括 HSTS、`X-Content-Type-Options: nosniff`、合理 CSP（尤其 `/memory/ui`）；
  具体设置应配合页面资源，不能照抄一个严格 CSP 后把 UI 自己挡掉。
- 关闭或保护不需要公开的 OpenAPI docs、管理接口和调试页面。

### 19.4 秘密和配置

`.env` 适合本地开发，不应提交真实密钥或烘焙进镜像。生产使用平台 Secret、Vault/KMS 或
容器编排的秘密注入，并做到：

- 每个环境使用不同 key；
- key 最小权限、可撤销、可轮换；
- 日志和错误响应永不打印秘密；
- 启动时校验必需配置，缺失就 fail fast；
- 配置模型化并记录非敏感配置版本。

### 19.5 优雅停机

容器收到 SIGTERM 后，理想流程是：

```text
readiness 变为 false
  → 停止接收新请求/新任务
  → 等待在途短请求完成
  → 当前队列任务到安全停止点，未开始任务留在持久队列
  → flush 日志/指标
  → 关闭 Graphiti、HTTP、Redis、数据库连接池
  → 在 termination grace period 内退出
```

当前 lifespan 的 `finally` 关闭 Graphiti 是正确基础，但没有“摘流量、drain 后台任务、任务
重新投递”这一整套协议。强制 kill 可能发生在任意写入步骤，所以仍必须依赖幂等和数据库约束。

### 19.6 部署参数没有万能公式

worker 数、线程池、连接池、Semaphore、队列长度都应通过压测决定：

1. 先确定下游 LLM RPM/TPM 和数据库连接上限；
2. 从小并发开始压测，记录 p95/p99、错误率、Token 和连接数；
3. 增加并发直到延迟陡升或 429/503 增多；
4. 把生产上限设在拐点之前并留故障余量；
5. 分别测试正常流量、突发、单用户滥用和下游变慢。

只看 CPU 核数套 `workers = 2 × cores + 1` 对 I/O 型 LLM 服务并不可靠。

## 20. 生产就绪分级检查表

### 20.1 当前成熟度

| 分类 | 当前状态 |
|------|----------|
| 当前已经实现 | 单 worker per-user 锁、L1+L2 数据隔离、基础进度幂等、Pydantic 基础模型、用量统计、简单 health、lifespan close |
| 文档已有方案但代码未实现 | API Key、Redis 分布式锁/进度/限流、asyncio.Queue/ARQ、任务状态查询、多 worker |
| 本次新增的生产要求 | body/批量上限、管理员封禁与取消、全局并发/背压、分层超时、readiness、统一错误、指标追踪、安全部署、系统测试 |

### 20.2 P0：公开给外部用户前必须完成

- [ ] API Key/JWT 认证，`user_id` 只由服务端凭证推导。
- [ ] 路由级授权；delete/admin 权限与普通读写权限分开。
- [ ] `messages/queries/qa` 数量、字符串长度、HTTP body 字节数上限。
- [ ] 每用户和每 IP 限流，超限返回 `429 + Retry-After`。
- [ ] LLM/数据库/接口总超时；明确断连后继续还是取消。
- [ ] 慢 ingest 改成 `202 + task_id + 有界持久队列`，或至少严格限制批量和并发。
- [ ] 全局昂贵调用并发上限；过载时快速 `503`，不能无限排队。
- [ ] 幂等键/唯一约束，验证任务重复投递不会重复写图。
- [ ] 管理员可禁用用户、撤销 queued 任务，并在安全点停止 processing 任务。
- [ ] liveness/readiness 分离，数据库不可用时停止接新流量。
- [ ] HTTPS、秘密不入仓库/镜像、管理接口不公开。
- [ ] 跨租户隔离自动化测试。

### 20.3 P1：稳定运营所需

- [ ] Redis 分布式锁和共享任务/限流状态，验证多 worker 正确性。
- [ ] HTTP/数据库连接池大小、获取 timeout、客户端缓存上限和淘汰关闭。
- [ ] 可重试错误分类、指数退避、最大次数、死信队列。
- [ ] 统一错误 schema、request_id、结构化脱敏日志。
- [ ] 请求/LLM/DB/队列核心指标和告警。
- [ ] OpenTelemetry 链路追踪。
- [ ] SIGTERM 优雅停机和部署 grace period。
- [ ] API 集成测试、依赖集成测试、压测基线和容量文档。
- [ ] Redis、FalkorDB、LLM 变慢或断开时的故障演练。

### 20.4 P2：规模化与安全加固

- [ ] 每用户 queued 上限、轮转或加权公平调度。
- [ ] Token/存储/请求月度配额、原子扣减和账单对账。
- [ ] 熔断与降级，例如 LLM 不可用时返回可解释错误或只做有限检索。
- [ ] WAF、安全头、严格 CORS、密钥轮换和审计日志。
- [ ] 多机部署、自动扩缩容、备份恢复和灾难演练。
- [ ] 数据保留/删除策略及隐私合规。
- [ ] SLO（可用性、p95 延迟）和错误预算。

### 20.5 最终判断

这份文档现在基本覆盖了生产级 FastAPI 接口应理解的**核心问题和设计检查项**，但“文档覆盖”
不等于“代码生产就绪”。真正达到生产级必须把 P0/P1 项逐项实现、测试、压测并持续监控。

对当前项目最合理的落地顺序：

```text
认证与可信 user_id
  → 单次请求/批量上限
  → 超时 + 全局并发上限
  → 202 task API + Redis/ARQ 有界队列
  → 每用户/IP 限流和管理员停用
  → Redis 分布式锁/幂等状态
  → readiness + 统一错误 + 指标日志
  → 多 worker 压测后上线
```

最后记住生产接口的总原则：

> 不信任客户端身份和输入；不允许任何等待、队列、缓存或连接无限增长；
> 不假设远程调用一定成功或取消；不在没有幂等的前提下重试写操作；
> 不把“服务还能返回 200”误当作“系统健康”。
