# 用户锁设计说明

本文说明 `main_server.py` 中 `app.state.user_locks` 和 `get_user_lock()` 的作用。

## 相关代码

服务启动时会创建两个应用级缓存：

```python
app.state.user_graphiti = {}
app.state.user_locks = {}
```

获取用户锁的函数：

```python
def get_user_lock(request: Request, user_id: str) -> asyncio.Lock:
    locks: dict[str, asyncio.Lock] = request.app.state.user_locks
    return locks.setdefault(user_id, asyncio.Lock())
```

它等价于：

```python
def get_user_lock(request: Request, user_id: str) -> asyncio.Lock:
    locks: dict[str, asyncio.Lock] = request.app.state.user_locks
    if user_id not in locks:
        locks[user_id] = asyncio.Lock()
    return locks[user_id]
```

也就是说，每个 `user_id` 会对应一把固定的异步锁：

```python
{
    'user_a': LockA,
    'user_b': LockB,
}
```

同一个用户的请求会拿同一把锁，不同用户的请求会拿不同的锁。

## 为什么要加用户锁

这个锁不是为了让不同用户互相隔离。不同用户本来就会使用不同的 `user_id`，并且这里还会给不同用户创建不同的 `Graphiti` 对象和 driver。

真正要保护的是：同一个用户的 `Graphiti + driver + 进度文件 + 图数据` 不要被多个请求同时操作。

## 原因一：保护用户 Graphiti 初始化

`get_user_graphiti()` 会按用户创建并缓存一个专属 `Graphiti` 对象：

```python
async def get_user_graphiti(request: Request, user_id: str) -> Graphiti:
    graphiti_by_user: dict[str, Graphiti] = request.app.state.user_graphiti
    if user_id in graphiti_by_user:
        return graphiti_by_user[user_id]

    base_graphiti = get_graphiti(request)
    driver = base_graphiti.driver.clone(database=user_id)
    graphiti = copy(base_graphiti)
    graphiti.driver = driver
    graphiti.clients = base_graphiti.clients.model_copy(update={'driver': driver})
    graphiti.nodes = NodeNamespace(driver, graphiti.embedder)
    graphiti.edges = EdgeNamespace(driver, graphiti.embedder)
    await graphiti.build_indices_and_constraints()
    graphiti_by_user[user_id] = graphiti
    return graphiti
```

如果同一个用户两个请求同时进来，并且这个用户还没有初始化过，可能出现：

```text
请求 A：发现 user_graphiti 中没有 alice
请求 B：也发现 user_graphiti 中没有 alice

请求 A：clone driver，copy Graphiti，创建索引
请求 B：也 clone driver，copy Graphiti，创建索引

请求 A：写入 user_graphiti['alice']
请求 B：再次写入 user_graphiti['alice']
```

这样会重复创建用户 driver、重复执行索引初始化，还可能导致缓存中最终保存的是后创建的对象。加锁后，同一个用户只会有一个请求进入初始化流程：

```text
请求 A：拿到 alice 的锁，创建并缓存 alice 的 Graphiti
请求 B：等待 alice 的锁
请求 A：释放锁
请求 B：再次检查缓存，直接复用已有 Graphiti
```

## 原因二：保护同一用户的图数据操作顺序

`Graphiti.add_episode()` 不是一条简单的数据库写入，它会做多步操作，例如：

```text
调用 LLM 抽取实体和关系
写入 episode
写入 entity node
写入 edge
执行去重
更新过期关系
维护图结构
```

这些步骤中间有很多 `await`。一个请求遇到 `await` 时会暂停，事件循环可以切换去执行另一个请求。

如果同一个用户两个写入请求同时执行，可能出现：

```text
请求 A：开始给 alice 添加消息 1
请求 A：await LLM/数据库，暂停
请求 B：开始给 alice 添加消息 2
请求 B：写入部分节点和边
请求 A：恢复，继续写入
```

这样不一定每次都会错，但会让同一个用户的图更新顺序变得不可控。对于知识图谱这种多步骤写入流程，保持同一用户内的顺序更稳。

## 原因三：保护注册进度文件

注册接口会读取和写入本地进度文件：

```python
group_progress = load_register_progress(group_id)
completed_episode_names = {item.episode_name for item in group_progress}
...
append_register_progress(group_id, progress_item)
```

如果同一个用户两个注册请求同时运行，可能出现：

```text
请求 A：读取进度文件，发现 msg_1 没处理
请求 B：也读取进度文件，发现 msg_1 没处理
请求 A：处理并追加进度
请求 B：也处理并追加进度
```

结果可能是重复处理、重复追加进度，或者进度和图数据库状态不一致。

加锁后，同一个用户的注册流程会串行执行，进度文件的读写顺序更清晰。

## 原因四：避免删除、查看、搜索和写入互相交错

代码中多个接口都会使用用户锁：

```python
async with get_user_lock(http_request, request.user_id):
    graphiti = await get_user_graphiti(http_request, request.user_id)
    ...
```

这些接口包括：

- `/memory/add`：写入记忆
- `/memory/delete`：删除该用户记忆
- `/memory/graph`：读取图视图
- `/memory/search`：搜索记忆
- `/memory/response`：基于搜索结果生成回答

加锁后，同一个用户的这些操作不会交错。例如不会出现：

```text
添加记忆写到一半
删除接口清空数据
添加接口继续写剩余数据
```

## 为什么不同用户还能并发

因为锁是按 `user_id` 存的：

```python
locks.setdefault(user_id, asyncio.Lock())
```

所以不同用户拿到的是不同锁：

```text
alice 请求使用 LockA
bob 请求使用 LockB
```

执行效果是：

```text
alice 的多个请求：排队
bob 的多个请求：排队
alice 和 bob：可以并发
```

这比使用一把全局锁更合理。如果用全局锁，`alice` 正在注册记忆时，`bob` 的搜索也要等待，会降低并发能力。

## 内测环境是否必须

如果内测流程能保证：

- 同一个用户不会同时发起多个请求
- 不会一边注册一边删除
- 不会重复提交同一批消息
- 服务只有一个进程

那么不加锁时也可能看起来正常。

但 HTTP 服务天然支持并发，请求也可能因为前端重复点击、脚本重试、超时重发、浏览器并发、批量评测等原因同时到达。加用户锁的成本很低，却能避免同一用户内部的初始化重复、进度文件竞争和图数据更新交错。

因此，这里的锁更像是一个按用户粒度的安全边界：

```text
同一用户：保证顺序
不同用户：允许并发
```

## 一句话总结

`user_locks` 的目的不是限制整个服务的并发，而是让同一个 `user_id` 共用的 `Graphiti`、driver、进度文件和图数据按顺序操作；同时保留不同用户之间的并发能力。
